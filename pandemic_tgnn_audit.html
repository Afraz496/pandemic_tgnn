Critical  Unbounded while(not stop) loop — can run forever
metalearn.py · lines 131–190  |  tl_base.py · lines 131–185
▶
Explanation
Both metalearn.py and tl_base.py wrap the entire training loop in while(not stop). The variable stuck is set to True when loss plateaus before epoch 30, which resets stop = False and causes the while-loop to re-enter and retrain from scratch. If the model is pathologically stuck (e.g., exploding gradients, all-zero data, wrong lr), stuck fires every single restart and the loop never exits. There is no maximum restart counter. On a GPU instance this burns quota indefinitely.
Proposed fix
Add a max_restarts guard and break unconditionally after it:
max_restarts = 3
restart_count = 0
while not stop:
    restart_count += 1
    if restart_count > max_restarts:
        print("Exceeded max restarts — forcing exit")
        stop = True
        break
    ...
Also log each restart with a timestamp so cloud billing spikes are visible.
2
Critical  retain_graph=True permanently leaks the entire computation graph
metalearn.py · line 27  |  tl_base.py · line 27
▶
Explanation
loss_train.backward(retain_graph=True) tells PyTorch never to free the intermediate activations stored during the forward pass. Every call to train() therefore leaks the full computation graph — all activations for every node in every batch remain pinned in GPU memory. The training loop calls train() hundreds of times per epoch, causing GPU memory to grow monotonically until OOM. On a CPU instance it will eventually swap-kill the process. This is the most dangerous single line in the repository.
Proposed fix
Remove retain_graph=True. It is only needed for exotic second-order gradient tricks (MAML-style), and the current code does not use the graph after backward():
loss_train.backward()  # not retain_graph=True
The MAML-style gradient accumulation in metalearn.py manually reads .grad.clone() after the step, which is safe without retaining the graph.

Critical  Model retrained from scratch at every test sample — O(N²) GPU calls
main.py · lines 57–90  |  metalearn.py · lines 124–195
▶
Explanation
The outer loop is for shift in range(1, ahead) (13 iterations) and the inner loop is for test_sample in range(start_exp, n_samples - shift) (~35–60 iterations per country). For each (shift, test_sample) pair, run_neural_model() trains a brand-new model for up to epochs=300. That gives roughly 13 × 50 × 300 × 4 countries = 780,000 epoch-level forward-backward passes. GPU instance time compounds fast. Nothing is cached between adjacent test samples even though each differs by only a single data point.
Proposed fix
Use warm-starting: save the checkpoint from test_sample=T and load it for test_sample=T+1 as the initial weights. Reduce epochs for warm restarts (e.g. 50 instead of 300). Add a dedicated CLI flag --warm-start so the behaviour is explicit. For metalearn, the eta checkpoint already exists — use it as the starting point for all test samples, not only the first.

Critical  LSTM cell state hardcoded to batch_size and pinned to CUDA at init
models.py · lines 193–196
▶
Explanation
self.cell is an nn.Parameter initialized at construction time with shape (nb_layers, batch_size, nhid) and immediately moved to CUDA via .cuda(). This hardwires the batch size into the model's parameters. When the final batch is smaller than batch_size (almost always — total samples rarely divide evenly), the forward pass hits a shape mismatch and falls into the bare except block with a silent slice. The explicit .cuda() also crashes on CPU-only machines (the Dockerfile installs the CPU-only PyTorch build).
Proposed fix
Remove the stateful cell parameter. Pass zero initial states at each forward call, respecting the actual batch dimension:
def forward(self, adj, features):
    features = features.view(self.window, -1, self.n_nodes)
    bsz = features.size(1)
    h0 = torch.zeros(self.nb_layers, bsz, self.nhid, device=features.device)
    c0 = torch.zeros(self.nb_layers, bsz, self.nhid, device=features.device)
    lstm_out, _ = self.lstm(features, (h0, c0))
    return self.linear(lstm_out[-1]).view(-1)
This also removes the try/except catch-all and the bare .cuda() call.

High  All batches materialized into GPU memory before training begins
neural_utils.py · lines 78–100  |  preprocess.py · lines 63–130
▶
Explanation
generate_batches() builds every batch tensor and .to(device)s all of them upfront. With MPNN_LSTM and graph_window=7, each batch element stacks 7 adjacency matrices and feature blocks. The full pre-allocated list can reach several hundred MB on GPU before training starts, and it is rebuilt fresh for every (shift, test_sample) pair — so this allocation/deallocation happens tens of thousands of times during a full run.
Proposed fix
Implement lazy batch generation: keep indices only, move tensors to device inside the training loop per batch. Or use a torch.utils.data.Dataset with pin_memory=True for async host→device transfers. At minimum, call del adj_train, features_train, y_train; torch.cuda.empty_cache() at the end of each run_neural_model call.

High  Early stopping uses integer rounding — can never fire on float losses
metalearn.py · line 170  |  tl_base.py · lines 155, 185
▶
Explanation
The early-stop check is len(set([round(val_e) for val_e in val_among_epochs[-50:]])) == 1. Because val_loss is cast to int (line 165: val_loss = int(...)), any validation loss between 0 and 1 always rounds to 0 or 1. When loss converges at, say, 0.3 the set is {0} and early stop fires instantly after 1 epoch. When loss oscillates between 0.3 and 0.9 the set is still {0} so early stop fires despite active learning. The early-stop logic is effectively broken for the expected loss range of this task (normalized case counts, typically <1).
Proposed fix
Remove the int() cast; store the float. Compare improvement delta instead of set cardinality:
PATIENCE = 50
best_val = float('inf')
patience_counter = 0
for epoch in range(epochs):
    ...
    if val_loss < best_val - 1e-4:
        best_val = val_loss
        patience_counter = 0
        torch.save(...)
    else:
        patience_counter += 1
    if patience_counter >= PATIENCE:
        break

High  nx.adjacency_matrix().toarray() called inside data loading — full dense N×N allocation per timestep
preprocess.py · lines 247, 268, 291, 316
▶
Explanation
gs_adj = [nx.adjacency_matrix(kgs).toarray().T for kgs in Gs] converts every graph to a dense NumPy array. For England (~150 NUTS-3 regions) this is a 150×150 float64 matrix per day × ~60 days = 1,350 dense matrices of 180 KB each. This is not catastrophic for current data, but it creates an O(N²) memory footprint per country per run and defeats any sparsity the mobility graphs might have. It also blocks GCNConv's native sparse path.
Proposed fix
Keep the adjacency as a sparse scipy matrix or convert directly to a torch sparse tensor. The downstream generate_batches call already uses sparse_mx_to_torch_sparse_tensor():
from preprocess import sparse_mx_to_torch_sparse_tensor
gs_adj = [sparse_mx_to_torch_sparse_tensor(
              nx.adjacency_matrix(kgs).T.astype('float32'))
          for kgs in Gs]
Then pass these tensors directly; remove the intermediate sp.block_diag(adj_tmp) conversion in generate_batches.

High  ARIMA trained per-region per-timestep — O(regions × timesteps) model fits
models.py · lines 46–67
▶
Explanation
The arima() function iterates over every region (labels.shape[0], ~100–150) for every test sample (~50), fitting a fresh ARIMA(1,0,0) via MLE at each step. That is 7,500+ statsmodels ARIMA fits per country. Each fit is slow (seconds on CPU). On 4 countries this is the dominant wall-clock cost — likely 30–60 min on a CPU instance.
Proposed fix
Vectorize with statsmodels.tsa.statespace.varmax or batch with pmdarima pipelines. Alternatively, precompute all ARIMA forecasts in a single sweep and cache them to disk before the neural model loop begins. AR(1) also has a closed-form OLS estimate that runs in microseconds per region.

High  generate_graphs reads one CSV per date from disk in a tight loop
preprocess.py · lines 360–375
▶
Explanation
Each call to generate_graphs(dates, country) opens and parses one CSV file per date — up to 63 files per country, 252 total. Each file is read via pd.read_csv and parsed into a NetworkX DiGraph by iterating row-by-row with d.iterrows(), which is notoriously slow. This function is called once at startup, but iterrows() on even a 1,000-edge graph can take seconds per graph.
Proposed fix
Replace iterrows() with vectorized edge loading:
for date in dates:
    d = pd.read_csv(f"graphs/{country}_{date}.csv", header=None,
                    names=['src','dst','w'])
    G = nx.from_pandas_edgelist(d, 'src', 'dst', 'w',
                                 create_using=nx.DiGraph())
Also consider pre-serializing all graphs to a single pickle or npz file per country to avoid repeated CSV parsing across runs.

High  train.py calls run_training() as a subprocess — duplicates process startup overhead per run
train.py · line 45
▶
Explanation
train.py spawns main.py via subprocess.run(["python", "main.py", ...]). This means environment initialisation, CUDA context setup, and all data loading happen in a child process that is thrown away after one run. If the cloud job scheduler retries on OOM or failure, the entire pipeline re-downloads data and restarts from scratch with no checkpointing of intermediate per-country results.
Proposed fix
Import and call experiment() directly from train.py, or use Python's multiprocessing if isolation is needed. Add per-country result caching: if results/results_{country}.csv already exists and is non-empty, skip that country.

Medium  No gradient clipping — training instability risk
neural_utils.py · line 21  |  metalearn.py · line 24
▶
Explanation
The MPNN_LSTM model stacks two GCNConv layers feeding two stacked LSTMs. This architecture is prone to exploding gradients through the temporal dimension, especially with sparse mobility graphs where some nodes have very high degree. No gradient clipping is applied before optimizer.step().
Proposed fix
loss_train.backward()
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
optimizer.step()

Medium  val_among_epochs grows without bound — accumulates all epoch losses
neural_utils.py · line 165  |  metalearn.py · line 166
▶
Explanation
val_among_epochs.append(val_loss) accumulates every epoch's validation loss in a Python list that is never pruned. Over 300 epochs × 650 (shift, test_sample) pairs per country, this creates 195,000-element lists per country. The list is also used in a set() conversion inside the epoch loop, which is O(early_stop/2) work per epoch — a hidden O(epochs²) term.
Proposed fix
Use a collections.deque(maxlen=early_stop) instead of a list. The early-stop check then becomes O(1):
from collections import deque
val_window = deque(maxlen=early_stop)
val_window.append(val_loss)
if len(val_window) == early_stop and len(set(round(v,2) for v in val_window)) == 1:
    break

Medium  os.chdir() used for data navigation — not thread-safe, side-effect on process
preprocess.py · lines 227–320
▶
Explanation
read_meta_datasets() navigates the filesystem by calling os.chdir("../Italy"), os.chdir("../Spain") etc. This mutates the process-wide working directory. If any exception is raised mid-function, the process is left in an unexpected directory. Any concurrent access (e.g., multiprocessing) will race on the cwd. The code relies on restoring cwd at the end with os.chdir("../../code") which is fragile.
Proposed fix
Pass absolute paths throughout and use pathlib.Path:
data_root = Path(__file__).parent.parent / "data"
for country_dir in [data_root / c for c in config['countries']]:
    labels = pd.read_csv(country_dir / label_file)
    graphs = generate_graphs(dates, country, graph_dir=country_dir / "graphs")

Medium  Checkpoint saved on every improvement — excessive disk I/O during training
neural_utils.py · lines 175–179  |  metalearn.py · lines 185–193
▶
Explanation
The checkpoint model_best.pth.tar is written to disk every epoch the validation loss improves. For early training (where loss decreases nearly every epoch), this means ~100 torch.save() calls per (shift, test_sample) pair. At ~180 KB per checkpoint × 650 pairs × 4 countries this is ~47 GB of total write amplification per run, which on GCS-backed volumes adds measurable cost and latency.
Proposed fix
Keep the best state dict in CPU RAM and save to disk only at the end of training:
best_state = None
if val_loss < best_val_acc:
    best_val_acc = val_loss
    best_state = {k: v.cpu().clone() for k, v in model.state_dict().items()}
# After loop:
torch.save({'state_dict': best_state}, 'model_best.pth.tar')

Medium  BUCKET_NAME read at module load without fallback — crashes entire container on missing env var
train.py · line 11
▶
Explanation
BUCKET_NAME = os.environ["BUCKET_NAME"] raises KeyError at import time if the environment variable is missing. More critically, client = storage.Client() and bucket = client.bucket(BUCKET_NAME) are also executed at module level, which initiates a GCS authentication round-trip before any training code runs. A misconfigured IAM policy or network hiccup silently blocks the entire job.
Proposed fix
Lazy-initialize the GCS client inside the functions that need it. Use os.environ.get("BUCKET_NAME") with a clear error message:
def get_bucket():
    name = os.environ.get("BUCKET_NAME")
    if not name:
        raise EnvironmentError("BUCKET_NAME env var is required")
    return storage.Client().bucket(name)

Medium  Deprecated nn.init.xavier_uniform (no trailing underscore) — silent wrong behavior
models.py · line 196
▶
Explanation
nn.init.xavier_uniform(tensor) (without underscore) is the deprecated form that returns a tensor but does not initialize in-place. In modern PyTorch it may be a no-op or raise a warning, leaving the cell state uninitialized (containing garbage values from the allocator). The correct call is nn.init.xavier_uniform_(tensor) (in-place).
Proposed fix
This is moot once the LSTM cell parameter is removed (see Issue 4). If a learnable initial state is still desired:
self.h0 = nn.Parameter(torch.zeros(nb_layers, 1, nhid))
nn.init.xavier_uniform_(self.h0)

Dockerfile · line 14  |  models.py · line 196
▶
Explanation
The Dockerfile pins torch==2.1.0 --index-url .../cpu, which has no CUDA support. The LSTM model calls .cuda() directly on the cell parameter. Running this container on a GPU instance will install CPU PyTorch and silently fall back to CPU, negating any GPU cost optimization. Proposed fix
Use a CUDA base image and the correct PyTorch CUDA wheel URL for GPU instances. Replace all hardcoded .cuda() calls with .to(device) where device is determined at runtime.
